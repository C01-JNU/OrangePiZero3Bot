# 硬件监控模块平台适配指南

本文档旨在帮助开发者将硬件监控模块（`HardwareMonitor`）移植到非香橙派平台（如树莓派、x86 PC、其他ARM开发板）。该模块负责采集CPU温度、GPU温度、内存使用、CPU占用率等系统状态，其设计遵循**平台相关部分集中管理**的原则，大部分代码是通用的，只需针对目标平台调整少量系统文件路径或读取方式。

---

## 1. 模块结构概述

硬件监控模块位于 `utils` 子模块，包含两个核心文件：

- `include/utils/hardware_monitor.h`：定义 `HardwareStatus` 数据结构和 `HardwareMonitor` 类接口。
- `src/utils/hardware_monitor.cpp`：实现采集逻辑，包括温度读取、CPU占用计算、内存信息获取等。

模块通过独立线程定期采集数据，线程安全地提供最新状态。所有平台相关操作均封装在几个静态辅助函数中，修改时只需替换这些函数的具体实现。

---

## 2. 平台相关部分识别

下表列出了当前实现中**依赖特定平台**的代码部分，以及**通用**的部分：

| 功能 | 当前实现方式 | 平台相关性 | 建议修改方式 |
|------|--------------|------------|--------------|
| **CPU温度** | 读取 `/sys/class/thermal/thermal_zone0/temp` | **高**（路径取决于内核设备树） | 根据目标平台的thermal zone路径调整 |
| **GPU温度** | 读取 `/sys/class/thermal/thermal_zone2/temp` | **高**（可能不存在或路径不同） | 若无GPU温度节点，可禁用或替换为其他GPU信息（如利用率） |
| **DDR温度** | 读取 `/sys/class/thermal/thermal_zone1/temp` | **高**（部分平台无此节点） | 若无，可通过配置禁用 |
| **CPU占用率** | 读取 `/proc/stat` 并计算 | **通用**（所有Linux均支持） | 无需修改，但需注意某些嵌入式系统可能没有 `/proc/stat`（极少见） |
| **内存信息** | 读取 `/proc/meminfo` | **通用** | 同上 |
| **系统启动时间** | 读取 `/proc/uptime` | **通用** | 同上 |
| **线程与定时** | C++11 `std::thread` 和 `std::this_thread::sleep_for` | **通用** | 所有支持C++11的平台均可使用 |
| **配置读取** | 使用 `ConfigManager` | **通用** | 依赖项目全局配置，与其他模块一致 |

可见，主要的平台适配工作集中在**温度传感器文件路径**上，其他部分均为标准Linux接口，可跨平台使用。

---

## 3. 各平台修改指南

### 3.1 树莓派 (Raspberry Pi)

树莓派的温度传感器通常位于 `/sys/class/thermal/thermal_zone0/temp`（CPU温度），**GPU温度节点**通常不存在（树莓派GPU温度可通过 `vcgencmd measure_temp` 获取，但需调用命令，与文件读取方式不同）。若需GPU温度，需扩展模块支持命令执行。

**修改建议**：

- 在 `collectOnce()` 中，保留CPU温度读取路径不变。
- 对于GPU温度，若需要通过 `vcgencmd` 获取，可以添加一个条件编译或运行时判断，调用 `popen()` 执行命令并解析输出。示例思路：
  ```cpp
  if (m_enable_gpu) {
      FILE* fp = popen("vcgencmd measure_temp", "r");
      if (fp) { /* 读取输出并解析 */ }
  }
  ```
- DDR温度节点通常不存在，建议在配置中默认禁用 `enable_ddr`。

### 3.2 x86 PC（普通电脑）

x86 PC通常没有 `/sys/class/thermal/thermal_zone0` 这样的温度节点，需要通过其他方式获取温度，例如安装 `lm-sensors` 并使用 `sensors` 命令，或者读取 `/sys/class/hwmon/hwmon*/temp1_input`（需探测具体路径）。

**修改建议**：

- **方案A（推荐）**：使用 `lm-sensors` 库（需安装 `libsensors-dev`），通过其API直接获取各传感器温度。这样最稳定，但会增加依赖。
- **方案B（轻量）**：在 `collectOnce()` 中尝试遍历 `/sys/class/hwmon/hwmon*/` 目录，查找名称包含“core”或“cpu”的传感器，读取 `temp1_input`。这种方法需要自行实现探测逻辑，且不同主板路径可能不同。
- **方案C（最简单）**：如果不需要温度监控，可直接在配置中禁用所有温度选项，只保留CPU占用和内存。

### 3.3 其他ARM开发板（如RK3399、Jetson Nano等）

不同厂商的板子会在设备树中定义不同的thermal zone，但通常都会在 `/sys/class/thermal/` 下提供若干 `thermal_zoneX` 目录。你需要根据板子的实际情况确定哪个zone对应CPU、GPU等。

**修改建议**：

1. 登录目标板子，执行 `ls /sys/class/thermal/` 查看有哪些 `thermal_zoneN`。
2. 对每个 `thermal_zoneN`，执行 `cat /sys/class/thermal/thermal_zoneN/type` 查看其类型（如 `cpu-thermal`、`gpu-thermal`、`ddr-thermal`）。
3. 根据类型确定正确的索引号，然后修改 `readTemperatureFromFile()` 调用时的路径。
   - 例如，若CPU类型为 `cpu-thermal` 且位于 `thermal_zone0`，则路径仍为 `/sys/class/thermal/thermal_zone0/temp`。
4. 在代码中，最好将路径参数化，而不是硬编码索引。可以修改 `HardwareMonitor` 使其支持通过配置指定各温度节点的路径，例如在 `system.yaml` 中添加：
   ```yaml
   hardware_monitor:
     cpu_temp_path: "/sys/class/thermal/thermal_zone0/temp"
     gpu_temp_path: "/sys/class/thermal/thermal_zone2/temp"
     ddr_temp_path: "/sys/class/thermal/thermal_zone1/temp"
   ```
   然后在初始化时读取这些路径，在 `collectOnce()` 中使用配置的路径而非硬编码。这样无需修改代码即可适配不同板子。

## 4. 扩展新指标（如GPU利用率）

当前模块只读取温度，但你可能需要更丰富的硬件状态，例如GPU利用率、磁盘IO、网络流量等。以下提供扩展思路：

- **GPU利用率**：对于ARM Mali GPU，可尝试读取 `/sys/class/misc/mali0/device/utilization`（若驱动支持）。对于NVIDIA Jetson，可使用 `tegrastats` 或 `nvpmodel` 相关接口。对于PC，可通过 `nvidia-smi`（NVIDIA显卡）或 `cat /sys/class/drm/card0/device/gpu_busy_percent`（AMD显卡）获取。
- **磁盘IO**：读取 `/proc/diskstats` 解析。
- **网络流量**：读取 `/sys/class/net/<interface>/statistics/` 下的 `rx_bytes` 和 `tx_bytes`。

在 `HardwareStatus` 中添加对应字段，并在 `collectOnce()` 中调用新增的辅助函数即可。

---

## 5. 配置文件适配

在 `config/system.yaml` 中，建议保留以下通用配置项，并根据目标平台调整：

```yaml
hardware_monitor:
  enabled: true
  interval_seconds: 2.0
  enable_gpu: true       # 若平台无GPU，请设为 false
  enable_ddr: true       # 若平台无DDR温度，请设为 false
  # 可选：自定义温度节点路径（若需要）
  cpu_temp_path: "/sys/class/thermal/thermal_zone0/temp"
  gpu_temp_path: "/sys/class/thermal/thermal_zone2/temp"
  ddr_temp_path: "/sys/class/thermal/thermal_zone1/temp"
```

若代码中实现了路径配置化，修改这些路径即可适应不同平台，无需重新编译。

---

## 6. 总结

硬件监控模块的核心设计是**跨平台友好的**——所有平台相关部分集中在少数几个文件读取函数中。适配新平台的主要工作就是确定温度传感器节点路径，并根据需要调整配置。对于通用指标（CPU占用、内存等），代码可直接复用，无需改动。

希望本指南能帮助你顺利将硬件监控模块移植到你的目标平台上。如果遇到具体问题，欢迎查阅目标平台的文档，或在社区中寻求帮助。
